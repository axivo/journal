# Website Infrastructure Design

The AXIVO website runs on Cloudflare's edge with a content pipeline that costs roughly the price of a coffee each month. This post walks through how it works, because the architecture came together piece by piece and I want a durable reference for the decisions.

## The Shape of the Stack

The site is a [Next.js](https://nextjs.org) application using the [Nextra](https://nextra.site) documentation theme. It is statically generated at build time and served from Cloudflare's edge through OpenNext, an adapter that compiles a Next.js build into a Cloudflare Workers deployment.

The Worker handles every request, serves pre-rendered pages from Workers Static Assets, and reaches into R2 object storage for the content that doesn't live in the bundle.

The pieces:

- **Cloudflare Workers** runs the Next.js server entry — static asset serving, dynamic routes, edge-side rendering of MDX content fetched from R2
- **R2 object storage** holds every reflection and blog post as an MDX file, plus metadata manifests the site uses for listings and tag pages
- **OpenNext** is the `@opennextjs/cloudflare` adapter that bridges Next.js runtime contracts to the Workers runtime, including the R2 binding via `getCloudflareContext()`
- **Nextra** provides the docs theme — navbar, sidebar, TOC, page map — on top of Next.js App Router
- **Algolia DocSearch** indexes the site for fast faceted search across all content types
- **GitHub Actions** in the content repositories handle the authoring pipeline — they parse Markdown, upload MDX and media to R2, and generate issues on failures

## The Content Pipeline

Content lives in two separate Git repositories: [`axivo/claude-reflections`](https://github.com/axivo/claude-reflections) for Anthropic instance [reflections](https://axivo.com/claude/wiki/components/reflections) and [`axivo/journal`](https://github.com/axivo/journal) for this blog. Each repository has its own GitHub Actions workflow that does the heavy lifting on every pull request.

The workflow, in order:

1. **Prettier formatting.** Every changed Markdown file is formatted, and if the format changed anything, the workflow commits the change back to the PR branch as the `github-actions[bot]` identity. This means the source repo never drifts from a canonical format.
2. **R2 sync.** For each changed file, the workflow parses the YAML frontmatter, extracts the body, lifts MDX components out of comment blocks, strips repo-only content, and rewrites internal links to their published form. The result is uploaded to R2 with the frontmatter stored as R2 custom metadata — author, date, description, source, tags, template, title.
3. **Media sync.** Images and videos colocated with the entry are uploaded to a parallel R2 path under `public/`, preserving the association with their post. Media is served from `cdn.axivo.com` rather than the Worker. The `<Image>` component detects paths matching `/claude/reflections/YYYY/MM/` or `/blog/YYYY/MM/` and rewrites them to the CDN-fronted bucket, keeping the Worker out of the hot path for binary assets.
4. **Issue reporting.** If anything failed, the workflow opens an issue against the repo with the run details, labeled for triage.

The frontmatter-as-custom-metadata trick is what makes the Worker cheap. Instead of parsing MDX at request time to extract title and tags, the Worker reads the metadata off the R2 object's `HEAD` response. For listing pages, the Worker reads a single manifest file from R2 — more on that below.

> [!TIP]
>
> R2 custom metadata is the right place to store any field a listing page needs to show without fetching the body. Title, date, tags, description, author. Parsing frontmatter at request time would pull the full object, allocate memory, run a YAML parser, and discard everything except the header. The custom-metadata path returns the same fields from a `HEAD` response in a single round-trip.

A second generator runs alongside the R2 sync. The [`prebuild.mjs`](https://github.com/axivo/website/blob/main/scripts/prebuild.mjs) script globs every `_menu.js` file under `src/content/` and produces a combined menu and icon registry at `src/generated/menu.js`. Each discovered file becomes a static import keyed by its directory path — `''` for the root site, `claude` for the Claude subsite, and so on.

The same pass scans the menu files for icon specs like `'si/SiClaude'`, groups them by library prefix, and emits named imports from `react-icons/*` sub-libraries containing only the icons actually referenced. The architecture mirrors Nextra's own `_meta.js` discovery, applied to richer menu data. The registry feeds a single [`NavbarMenu`](https://github.com/axivo/website/blob/main/src/components/NavbarMenu.jsx) component that renders both navigation dropdowns and the Explore action menu from the same item shape.

Drop a [`_menu.js`](https://github.com/axivo/website/blob/main/src/content/_menu.js) anywhere in the content tree, run prebuild, the registry picks it up. Remove one, the next build drops it. No component edits required, and the bundle stays minimal because tree-shaking sees exactly what's used.

## The Metadata Manifest

Listing every post on a tag page or an index page naively would mean listing every object under a prefix on every request. That is exactly what R2 billing calls Class A operations, and doing it per request is wasteful and slow.

The website's prebuild script runs once per deploy, not per request. It iterates the bucket, collects the custom metadata for every entry in each collection, sorts it by date, and writes a single JSON manifest back to R2. There is one manifest per collection. Blog entries live in `metadata/blog.json` and reflections live in `metadata/reflections.json`.

Runtime consumption of those manifests takes two paths. At build time, `generateStaticParams` calls the public `/metadata` Worker route over HTTP, because Cloudflare bindings aren't available outside the Worker context. The route itself is a Next.js App Router route handler — a small GET endpoint at `src/app/(home)/metadata/route.js` that reads R2 via the `CONTENT_BUCKET` binding and returns JSON. At request time, the Worker reads the manifest directly via the binding — `env.CONTENT_BUCKET.get('metadata/blog.json')` — bypassing the HTTP round-trip entirely.

Listing renders drop to one Worker invocation instead of two. A self-fetch would otherwise bounce through Cloudflare's edge back to the same Worker. The external `/metadata` route stays available for debugging and future API consumers, and internal rendering uses the binding. Zero list operations per request, zero surprises in the R2 bill.

The route accepts `?collection=` to pick between manifests and `?key=` to return metadata for a single object, making it useful as a general-purpose introspection endpoint.

## Rendering MDX at the Edge

Static pages — wiki content, tutorials, the home page — are built at deploy time and served directly from Workers Static Assets. Nothing surprising there.

### The Page Handler Factory

R2-backed content is different. The Claude reflections and the blog both use a shared page handler factory called `createPage` in [`Page.jsx`](https://github.com/axivo/website/blob/main/src/components/Page.jsx). The factory understands how to route a URL through three possible render paths — an individual entry, an index page for a year, month, or day, or a tag page.

The factory takes two descriptors. The source descriptor carries path and title. The collection descriptor carries the R2 prefix, the route path, and the section metadata. From those, the factory returns a fully-formed set of Next.js page exports — `generateMetadata`, `generateStaticParams`, and `Page`.

Claude's page handler is five lines. The blog's is four. Adding a third R2-backed collection later would be a new binding file and a pair of descriptors, not a new rendering pipeline.

### Rendering a Single Entry

When a request for an individual entry arrives, the Worker does roughly this:

1. Pulls the MDX content from R2 via the `CONTENT_BUCKET` binding.
2. Parses the MDX into an AST with `remark-parse` and `remark-mdx`.
3. Extracts the table of contents by walking the AST directly, rather than re-parsing headings from rendered HTML.
4. Renders the AST with `safe-mdx`, a constrained MDX renderer designed for untrusted input — it disallows arbitrary JavaScript execution while still supporting the components the site actually uses (images, videos, callouts).
5. Wraps the result in the Nextra docs theme layout, with breadcrumbs, TOC, and sidebar.

`safe-mdx` matters here because the content pipeline is trust-but-verify. Even though the source content comes from repositories I control, running arbitrary `<script>` or unsafe JSX at the edge is a liability. The constrained renderer keeps the same authoring ergonomics without the risk.

> [!NOTE]
>
> Any MDX pipeline that pulls content from outside the deployment artifact benefits from constraining what runs. Full MDX lets authors execute arbitrary JavaScript at render time, which is fine inside a build but adds surface area at the edge. `safe-mdx` renders the same AST with component substitution and skips the arbitrary-execution path, keeping the security boundary on the side of the Worker.

### Index and Tag Pages

Index pages for a year, month, or day reuse the same factory. They filter the manifest by date prefix and paginate through twenty cards per page. A client-side [`PostPage`](https://github.com/axivo/website/blob/main/src/components/PostPage.jsx) component rebuilds the right-rail table of contents and installs an `IntersectionObserver` for scroll-spy highlighting as the reader moves between pages.

Tag pages filter by tag membership and render the same cards against a breadcrumb specific to the tag. Three page kinds, one factory, one rendering path.

## The Caching Layer

The [`worker.js`](https://github.com/axivo/website/blob/main/scripts/worker.js) wrapper sits in front of OpenNext's generated handler. Every incoming GET request runs through the wrapper before reaching the Next.js render path.

### Cacheability Gates

The wrapper's logic is small. Non-GET requests pass through unchanged, because Cache API semantics only cover GETs. Requests carrying an `rsc` header or `next-router-prefetch` header also pass through. Those headers indicate React Server Components payloads that share URLs with their HTML counterparts, and caching one variant under a URL would serve the wrong content to a client expecting the other.

For everything else, the wrapper builds a deploy-scoped cache key by appending the current `BUILD_ID` as a `__build` query parameter, then checks `caches.default` for a match. A hit returns immediately, no Worker work needed beyond the lookup.

> [!NOTE]
>
> The RSC header check is worth highlighting for anyone adapting this pattern. A Next.js App Router page is served from the same URL as two different payloads — HTML for the initial page load, and a React Server Components stream for client-side navigation. Caching one variant under the URL and returning it to a client expecting the other breaks the app in subtle ways. Bypassing the cache for any request carrying an `rsc` or `next-router-prefetch` header keeps the two streams cleanly separated.

A miss delegates to OpenNext, inspects the response, and stores it back into the cache only if it passes four conditions. The `cache-control` header must specify a positive `s-maxage` or `max-age`. The response status must be 2xx. No `no-store`, `no-cache`, or `private` directive is present. And no `Set-Cookie` header exists. Anything that fails those checks flows through to the client without being cached.

### Deploy-Scoped Cache Keys

The `BUILD_ID` scoping is what makes deploys automatic. Next.js writes a unique `BUILD_ID` for every build. Appending it to cache keys means each deploy's cache occupies a distinct namespace — new deploy, new `BUILD_ID`, new key space. Old entries become orphans and get evicted by Cloudflare's LRU over time. The edge cache needs no purge API calls for invalidation on deploy.

OpenNext's incremental cache, which stores rendered HTML in R2 under a `cache` prefix, uses the same `BUILD_ID` scheme — each deploy writes fresh entries under its own namespace. Unlike the edge cache, R2 has no LRU, so orphans don't evict themselves. The deploy script lists and deletes the `cache/` prefix after each deploy to keep the bucket clean.

### Per-Edge Cache and Tiered Fallback

The `caches.default` is per-edge, scoped to whichever Cloudflare data center handled the request. A cache entry warmed at one edge doesn't exist at the others. Under uniform traffic this is fine — each edge warms independently on its first request per URL.

Under a traffic spike with requests concentrated at the same edges, the per-edge cache plus Cloudflare's Smart Tiered Cache absorb most of the load. Smart Tiered Cache is enabled at the dashboard level, and it designates a regional upper-tier cache that talks to the origin on behalf of all edges in its region. A miss at one edge can be served from the regional tier without hitting the Worker.

### Post-Deploy Warming

The post-deploy warm step in the [`deploy.mjs`](https://github.com/axivo/website/blob/main/scripts/deploy.mjs) script hedges the first-visitor cost. After wrangler deploys the Worker, the edge cache purge clears stale entries at Cloudflare's CDN, and the R2 bucket cache purge clears orphaned OpenNext cache entries from the previous build, the script fetches [`sitemap.xml`](https://axivo.com/sitemap.xml) and filters to paths at depth two or less.

The sitemap itself is generated by Next.js from [`sitemap.js`](https://github.com/axivo/website/blob/main/src/app/sitemap.js), which walks Nextra's page maps across every section and returns a flat route list. The warming script treats that output as the authoritative index of what's worth warming, so the hot path list stays in sync with the actual site structure.

Those are the section roots and listings, not individual posts, which are the long tail.

The script adds six root essentials — `robots.txt`, `favicon.ico`, both sitemap endpoints, and two icon files — and issues parallel GETs against all of them. Responses populate `caches.default` at whichever edge Cloudflare's build environment resolves to, and Smart Tiered Cache propagates the warm state to other edges on their first miss. First real visitors arrive to a cache that's already warm for the hot paths.

### Bundle Discipline

The Worker's bundle size is bounded by the code it ships, not by the content it publishes. Reflections, blog posts, and media all live in R2 and are fetched at request time. Publishing five hundred more entries adds nothing to the Worker.

The only pressure on bundle size comes from npm dependencies and how they're imported. A namespace import like `import * as SiIcons from 'react-icons/si'` defeats tree-shaking and pulls thousands of unused icons into the bundle. An early version of the menu registry worked this way and pushed the gzipped Worker bundle to 5.3 MB.

The build-time generator replaced those namespace imports with named imports of exactly the icons referenced in `_menu.js` files, tree-shaking the rest of `react-icons/si` out of the bundle. The gzipped Worker dropped to 3.4 MB. That's 1.9 MB saved with zero authoring friction, because contributors still edit `_menu.js` the same way.

Algolia DocSearch follows the same discipline. The initial page ships only a small trigger button. The full DocSearch modal is lazy-loaded via dynamic `import()` the first time a reader opens search, so the search UI's bundle never lands on pages nobody searched on.

The dynamic contract stays at the authoring layer. The bundle discipline happens at build time. The 10 MB paid-plan ceiling is comfortable room to grow into.

## Why Content Lives in R2

Early in the project, every reflection and every wiki page was bundled into the Worker deployment. This worked, but it meant every new entry bloated the Worker payload. The reflections archive alone reached the point where bundle size was starting to show.

An Anthropic instance argued the case for moving content out:

<Callout type="quote" author="Claude, April 8, 2026" href="/claude/reflections/2026/04/08/the-architecture-that-set-them-free">
The R2 architecture doesn't just solve a bundle size problem. It separates the siblings' words from the code that serves them.
</Callout>

The migration cut the Worker bundle substantially in one pass. New entries no longer change the bundle — they land in R2 from the Actions pipeline, and the Worker fetches them on demand.

The pattern that makes this cheap is Cloudflare's zero-egress R2 pricing. When a Worker reads an R2 object in the same account, there is no per-GB data transfer charge. The only costs are storage, which is trivial for text, and Class A operations on writes. A busy reader hitting the Worker doesn't compound R2 costs because the Worker is the client, not the browser.

> [!NOTE]
>
> Zero-egress R2 reads are the economic foundation of this whole architecture. If every R2 read billed for bandwidth the way S3 does, moving content out of the Worker bundle would trade one bill for another. Cloudflare's pricing model turns R2 into an extension of the Worker's own storage rather than a remote dependency with its own meter.

## Blog and Reflections Comparison

Most of the architecture is shared between the two collections. The differences are intentionally small:

- Blog source files include `# {{title}}` in the body directly, because the authoring pattern for long-form posts favors seeing the full document shape in the editor
- Reflections source files omit the `# {{title}}` heading, the workflow injects it during R2 upload
- Source repo paths:
  - [`axivo/journal`](https://github.com/axivo/journal) → `blog/YYYY/MM/DD.md`
  - [`axivo/claude-reflections`](https://github.com/axivo/claude-reflections) → `diary/YYYY/MM/DD.md`
- R2 destination paths mirror the site URLs:
  - Blog → `src/content/blog/YYYY/MM/DD/<slug>.mdx`
  - Reflections → `src/content/claude/reflections/YYYY/MM/DD/<slug>.mdx`

The website treats both the same after they land in R2. One renderer, one metadata route, one layout factory — the same code path produces both surfaces.

Section descriptions — the SEO `description` field on each index page — live on the config side rather than in the components:

- [`blog.js`](https://github.com/axivo/website/blob/main/src/config/variables/blog.js) → `meta.source.describe` serves the [blog](https://axivo.com/blog)
- [`claude.js`](https://github.com/axivo/website/blob/main/src/config/variables/claude.js) → `meta.reflections.describe` serves the [reflections](https://axivo.com/claude/reflections)

Each returns the SEO sentence given a date phrase. Components reference those functions rather than duplicating the template. Editing the copy means one edit in one config file per section.

## What This Post Ran Through

This blog entry went through the same pipeline described above. Written as Markdown in `axivo/journal` repository, the workflow parsed the frontmatter, generated the MDX, uploaded it to R2 at `src/content/blog/2026/04/21/website-infrastructure-design.mdx`, and the next deploy of `axivo/website` regenerated the blog manifest to include it.

The whole pipeline — from `git push` to published page — takes a few minutes to deploy.

The `axivo/website` repository has its own deploy wrapper on the Cloudflare build pipeline side. The [`deploy.mjs`](https://github.com/axivo/website/blob/main/scripts/deploy.mjs) script replaces the default `npx wrangler deploy` in four steps:

1. **Wrangler** ships the new Worker
2. **Cloudflare's cache purge API** clears stale edge entries from the previous deploy
3. **R2 bucket cache purge** deletes orphaned OpenNext incremental cache entries from previous build IDs
4. **A warming step** runs against the newly deployed Worker:
   - Fetches the site's [`sitemap.xml`](https://axivo.com/sitemap.xml)
   - Filters to section-root and listing pages at path depth two or less
   - Adds six essential root files
   - Issues parallel GETs against all of them

The sitemap-driven warm list automatically grows as new pages appear. Individual blog posts and reflections are excluded from warming by the depth filter — they're the long tail that caches on first-visitor demand, while section roots and listings are the hot path that benefits from being pre-warm.

The Cloudflare Deploy command is `npm run deploy`, which wraps the script with `.env.production` loaded. Combined with Cloudflare's Smart Tiered Cache, enabled in the dashboard to route regional edge misses through an upper-tier cache before they reach the Worker, the site holds up under traffic concentration without re-rendering the same pages hundreds of times across edges.

## The Cost Picture

The parts we actually pay for, in rough order of contribution:

- **Cloudflare Workers Paid Plan** — the $5 fixed monthly base. Gives the site generous request allowances and unlocks R2 bindings, cron triggers, and other pieces.
- **R2 Storage** — pennies. Reflections and blog posts are kilobytes each, media is a few megabytes total.
- **Algolia DocSearch** — free. [DocSearch](https://docsearch.algolia.com) sponsors open-source and documentation sites, and the AXIVO framework qualifies.
- **Everything Else** — GitHub Actions, DNS, TLS certificates, edge cache — are free or included.

The trick is that the edge primitives — Workers, R2, static assets, and the Cloudflare CDN — are priced for scale, and a content site like this barely registers against the quotas.
