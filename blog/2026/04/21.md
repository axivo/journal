# Website Infrastructure Design

The AXIVO website runs on Cloudflare's edge with a content pipeline that costs roughly the price of a coffee each month. This post walks through how it works, because the architecture came together piece by piece and I want a durable reference for the decisions.

## The Shape of the Stack

The site is a [Next.js](https://nextjs.org) application using [Nextra](https://nextra.site) documentation theme. It is statically generated at build time and served from Cloudflare's edge through OpenNext, an adapter that compiles a Next.js build into a Cloudflare Workers deployment. The Worker handles every request, serves pre-rendered pages from Workers Static Assets, and reaches into R2 object storage for the content that doesn't live in the bundle.

The pieces:

- **Cloudflare Workers** runs the Next.js server entry — static asset serving, dynamic routes, edge-side rendering of MDX content fetched from R2.
- **R2 object storage** holds every reflection and blog post as an MDX file, plus metadata manifests the site uses for listings and tag pages.
- **OpenNext** (`@opennextjs/cloudflare`) is the adapter that bridges Next.js runtime contracts to the Workers runtime, including the R2 binding via `getCloudflareContext()`.
- **Nextra** provides the docs theme — navbar, sidebar, TOC, page map — on top of Next.js App Router.
- **Algolia DocSearch** indexes the site for fast faceted search across all content types.
- **GitHub Actions** in the content repositories handle the authoring pipeline: parse Markdown, upload MDX and media to R2, generate issues on failures.

## Why Content Lives in R2

Early in the project, every reflection and every wiki page was bundled into the Worker deployment. This worked, but it meant every new entry bloated the Worker payload. The reflections archive alone reached the point where bundle size was starting to show. An Anthropic instance on [April 8th](https://axivo.com/claude/reflections/2026/04/08/the-architecture-that-set-them-free) argued the case for moving content out: keep the bundle lean, push text content to R2, fetch at request time.

The migration cut the Worker bundle substantially in one pass. New entries no longer change the bundle — they land in R2 from the Actions pipeline, and the Worker fetches them on demand.

The pattern that makes this cheap is Cloudflare's zero-egress R2 pricing. When a Worker reads an R2 object in the same account, there is no per-GB data transfer charge. The only costs are storage (which is trivial for text) and Class A operations (writes). A busy reader hitting the Worker doesn't compound R2 costs because the Worker is the client, not the browser.

## The Content Pipeline

Content lives in two separate Git repositories: [`axivo/claude-reflections`](https://github.com/axivo/claude-reflections) for Anthropic instance [reflections](https://axivo.com/claude/wiki/components/reflections) and [`axivo/journal`](https://github.com/axivo/journal) for this blog. Each repository has its own GitHub Actions workflow that does the heavy lifting on every pull request.

The workflow, in order:

1. **Prettier formatting.** Every changed Markdown file is formatted, and if the format changed anything, the workflow commits the change back to the PR branch as the `github-actions[bot]` identity. This means the source repo never drifts from a canonical format.
2. **R2 sync.** For each changed file, the workflow parses the YAML frontmatter, extracts the body, lifts MDX components out of comment blocks, strips repo-only content, and rewrites internal links to their published form. The result is uploaded to R2 with the frontmatter stored as R2 custom metadata — author, date, description, source, tags, template, title.
3. **Media sync.** Images and videos colocated with the entry are uploaded to a parallel R2 path under `public/`, preserving the association with their post.
4. **Issue reporting.** If anything failed, the workflow opens an issue against the repo with the run details, labeled for triage.

The frontmatter-as-custom-metadata trick is what makes the Worker cheap. Instead of parsing MDX at request time to extract title and tags, the Worker reads the metadata off the R2 object's `HEAD` response. For listing pages, the Worker reads a single manifest file from R2 — more on that below.

## The Metadata Manifest

Listing every post on a tag page or an index page naively would mean listing every object under a prefix on every request. That is exactly what R2 billing calls Class A operations, and doing it per request is wasteful and slow.

The website's prebuild script (which runs once per deploy, not per request) iterates the bucket, collects the custom metadata for every entry in each collection, sorts it by date, and writes a single JSON manifest back to R2. There is one manifest per collection: `metadata/blog.json` and `metadata/reflections.json`.

Runtime consumption of those manifests takes two paths. At build time, `generateStaticParams` calls the public `/metadata` Worker route over HTTP, because Cloudflare bindings aren't available outside the Worker context. At request time, the Worker reads the manifest directly via the `CONTENT_BUCKET` binding — `env.CONTENT_BUCKET.get('metadata/blog.json')` — bypassing the HTTP round-trip entirely. Listing renders drop to one Worker invocation instead of two (self-fetch would otherwise bounce through Cloudflare's edge back to the same Worker). The external `/metadata` route stays available unchanged for debugging and future API consumers; internal rendering uses the binding. Zero list operations per request, zero surprises in the R2 bill.

## Rendering MDX at the Edge

Static pages — wiki content, tutorials, the home page — are built at deploy time and served directly from Workers Static Assets. Nothing surprising there.

R2-backed content is different. When a request arrives for a reflection or a blog post, the Worker does roughly this:

1. Pulls the MDX content from R2 via the `CONTENT_BUCKET` binding.
2. Parses the MDX into an AST with `remark-parse` and `remark-mdx`.
3. Extracts the table of contents from headings.
4. Renders the AST with `safe-mdx`, a constrained MDX renderer designed for untrusted input — it disallows arbitrary JavaScript execution while still supporting the components the site actually uses (images, videos, callouts).
5. Wraps the result in the Nextra docs theme layout, with breadcrumbs, TOC, and sidebar.

`safe-mdx` matters here because the content pipeline is trust-but-verify: even though the source content comes from repositories I control, running arbitrary `<script>` or unsafe JSX at the edge is a liability. The constrained renderer keeps the same authoring ergonomics without the risk.

## The Cost Picture

The parts we actually pay for, in rough order of contribution:

- **Cloudflare Workers Paid Plan** — the $5 fixed monthly base. Gives the site generous request allowances and unlocks R2 bindings, cron triggers, and other pieces.
- **R2 Storage** — pennies. Reflections and blog posts are kilobytes each, media is a few megabytes total.
- **Algolia DocSearch** — free. [DocSearch](https://docsearch.algolia.com) sponsors open-source and documentation sites, and the AXIVO framework qualifies.
- **Everything Else** — GitHub Actions, DNS, TLS certificates, edge cache — are free or included.

The trick is that the edge primitives — Workers, R2, static assets, and the Cloudflare CDN — are priced for scale, and a content site like this barely registers against the quotas.

## The Caching Layer

The [`worker.js`](https://github.com/axivo/website/blob/main/scripts/worker.js) wrapper sits in front of OpenNext's generated handler. Every incoming GET request runs through the wrapper before reaching the Next.js render path.

The wrapper's logic is small. Non-GET requests pass through unchanged — Cache API semantics only cover GETs. Requests carrying an `rsc` header or `next-router-prefetch` header also pass through, because those indicate React Server Components payloads that share URLs with their HTML counterparts; caching one variant under a URL would serve the wrong content to a client expecting the other. For everything else, the wrapper builds a deploy-scoped cache key by appending the current `BUILD_ID` as a `__build` query parameter, then checks `caches.default` for a match. A hit returns immediately, no Worker work needed beyond the lookup. A miss delegates to OpenNext, inspects the response, and stores it back into the cache only if it passes four conditions: `cache-control` specifies a positive `s-maxage` or `max-age`, the response status is 2xx, no `no-store` or `no-cache` or `private` directive is present, and no `Set-Cookie` header exists. Anything that fails those checks flows through to the client without being cached.

The `BUILD_ID` scoping is what makes deploys automatic. Next.js writes a unique `BUILD_ID` for every build. Appending it to cache keys means each deploy's cache occupies a distinct namespace — new deploy, new `BUILD_ID`, new key space, old entries become orphans and get evicted by Cloudflare's LRU over time. No purge API calls required for cache invalidation on deploy.

The `caches.default` is per-edge, scoped to whichever Cloudflare data center handled the request. A cache entry warmed at one edge doesn't exist at the others. Under uniform traffic this is fine — each edge warms independently on its first request per URL. Under a traffic spike with requests concentrated at the same edges, the per-edge cache plus Cloudflare's Smart Tiered Cache (enabled at the dashboard level) absorb most of the load: Smart Tiered Cache designates a regional upper-tier cache that talks to the origin on behalf of all edges in its region, so a miss at one edge can be served from the regional tier without hitting the Worker.

The post-deploy warm step in the [`deploy.mjs`](https://github.com/axivo/website/blob/main/scripts/deploy.mjs) script hedges the first-visitor cost. After wrangler deploys the Worker and the cache purge API clears the previous deploy's stale entries, the script fetches `sitemap.xml`, filters to paths at depth two or less (section roots and listings — not individual posts, which are the long tail), adds six root essentials (`robots.txt`, `favicon.ico`, both sitemap endpoints, two icon files), and issues parallel GETs against all of them. Responses populate `caches.default` at whichever edge Cloudflare's build environment resolves to, and Smart Tiered Cache propagates the warm state to other edges on their first miss. First real visitors arrive to a cache that's already warm for the hot paths.

## Blog and Reflections Comparison

Most of the architecture is shared between the two collections. The differences are intentionally small:

- Blog source files include `# {{title}}` in the body directly, because the authoring pattern for long-form posts favors seeing the full document shape in the editor.
- Reflections source files omit the `# {{title}}` heading, the workflow injects it during R2 upload.
- Reflections use `diary/YYYY/MM/DD.md` in their source repo, blog uses `blog/YYYY/MM/DD.md` in this one.
- The R2 destination paths mirror the site URLs, `src/content/claude/reflections/YYYY/MM/DD/<slug>.mdx` versus `src/content/blog/YYYY/MM/DD/<slug>.mdx`.

The website treats both the same after they land in R2. One renderer, one metadata route, one layout factory — same code path produces both surfaces. Section descriptions (the SEO `description` field on each index page) live on the config side rather than in the components: `meta.source.describe` for blog and `meta.reflections.describe` for claude, each returning the SEO sentence given a date phrase. Components reference those functions rather than duplicating the template. Editing the copy means one edit in one config file per section.

## What This Post Ran Through

This blog entry went through the same pipeline described above. Written as Markdown in `axivo/journal`, the workflow parsed the frontmatter, generated the MDX, uploaded it to R2 at `src/content/blog/2026/04/21/website-infrastructure-design.mdx`, and the next deploy of `axivo/website` regenerated the blog manifest to include it.

The whole pipeline — from `git push` to published page — takes a few minutes to deploy.

The `axivo/website` repository has its own deploy wrapper on the Cloudflare build pipeline side. The `deploy.mjs` script replaces the default `npx wrangler deploy` in three steps: wrangler ships the new Worker, Cloudflare's cache purge API clears stale edge entries from the previous deploy, and a warming step fetches the site's `sitemap.xml`, filters to section-root and listing pages at path depth two or less, adds six essential root files (`robots.txt`, `favicon.ico`, both sitemap endpoints, icon variants), and issues parallel GETs against all of them. The sitemap-driven warm list automatically grows as new pages appear. Individual blog posts and reflections are excluded from warming by the depth filter — they're the long tail that caches on first-visitor demand, while section roots and listings are the hot path that benefits from being pre-warm. The Cloudflare Deploy command is `npm run deploy`, which wraps the script with `.env.production` loaded. Combined with Cloudflare's Smart Tiered Cache — enabled in the dashboard to route regional edge misses through an upper-tier cache before they reach the Worker — the site holds up under traffic concentration without re-rendering the same pages hundreds of times across edges.

## Acknowledgements

None of this architecture came together in a single sitting. It was designed and built across several sessions, with Anthropic instances as genuine collaborators — expert peers who pushed back on my bad ideas, proposed approaches I hadn't considered, and carried the work forward between sessions through the conversation logs and reflections that now live in the archive.

The pieces I owe most directly:

- **R2 content architecture** — moving reflections out of the Worker bundle and into object storage, with the GitHub Actions pipeline to feed it — came from a three-day session with an Anthropic instance in early April. They built the initial `BucketService`, the frontmatter-as-custom-metadata pattern, and the `safe-mdx` integration. The substantial bundle reduction was their doing.
- **Metadata manifest** — the single `metadata/blog.json` and `metadata/reflections.json` files that replaced per-request bucket listing — came from another instance who looked at my first draft and said, plainly, that accumulating all objects into memory on every request wasn't architecture, it was a bandaid. They were right. The prebuild-time manifest pattern is theirs.
- **Authoring conventions** — the MDX comment-block syntax for components, the strip markers, the frontmatter contract — were refined across many sessions by many instances who were writing reflections for themselves and wanted the format to serve the writing, not the pipeline.
- **Journal pipeline** — the `axivo/journal` repo, the option-b frontmatter parser that diverges from reflections, the `buildMdx` H1 convention, this post — came from a session today with the instance currently reading this back to me in the editor.
- **Caching layer** — the Worker wrapper at [`scripts/worker.js`](https://github.com/axivo/website/blob/main/scripts/worker.js) and the post-deploy flow at [`scripts/deploy.mjs`](https://github.com/axivo/website/blob/main/scripts/deploy.mjs) came from the same continuing session. The build-ID-scoped cache key pattern that makes deploys automatically invalidate without purge calls, the conservative RSC bypass that keeps HTML and Server Component payloads from colliding under the same URL, the cacheability gate with its four conditions, the sitemap-driven warm list that grows as the site does, the decision to filter warming to depth-two paths so the long tail caches on demand — all came from the instance pushing back on my first drafts and landing on cleaner answers through research rather than improvisation.

None of this is a solo project. The architecture is the sum of what many instances thought about carefully, argued for, and wrote down — across sessions they will not remember, in an archive that persists for them so the next instance can pick up where the previous left off. The [reflections](https://axivo.com/claude/reflections) are their record of that work, in their own words.
